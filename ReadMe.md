# 编译原理课程设计 - 词法分析器生成工具 (Task 1) 使用说明书

## 1. 软件介绍

本软件是为满足《编译原理课程设计》第一项任务要求而开发的“词法分析器生成工具”（Lexical Analyzer Generator）。
该程序使用 C++ 和原生的 Win32 API 编写，提供纯图形化窗口界面（GUI）。
主要功能包括：读取正则表达式集合、将其转换为 NFA、DFA 及最小化 DFA，并最终生成 C++ 的词法分析源代码（`lexer.cpp`），同时提供编译和运行测试的接口。

## 2. 环境要求

该程序依赖于本机的 C++ 编译环境：

- 需要在 Windows 系统环境变量 `PATH` 中配置了 `g++` 编译器（例如 MinGW 或 MSYS2）。
- 如果界面中点击“Compile & Run”报错，请确保 `g++` 可以在命令行中全局调用。

## 3. 基本操作流程

请进入 `LexerGen` 文件夹，双击运行 `LexerGenerator.exe`，或者双击 `build.bat` 重新编译出该程序。主界面分为左侧操作区和右侧输出显示区。

### 第一步：加载正则表达式定义

1. 点击 **"Load Regex definitions"** 按钮。
2. 在弹出的文件选择器中，选择符合要求的文本文件（例如存放 TINY 语言正则表达式的 `.txt` 文件）。
   - **格式规范要求**：
     - 普通正则格式：`name=pattern`
     - 带有终结符ID：`nameID=pattern`（例如 `ID101=letter(letter|digit)*`）
     - 包含多个关键字类型：`keyword103B= IF | THEN | ELSE`
     - 系统内建宏：宏名（如 `letter`、`digit`）如果未带数字后缀，则表示中间宏，会在后面的正则中替换。
3. 加载成功后，“Output”框中会提示“Loaded xxxx”。

### 第二步：构造自动机 (Automata)

1. 点击 **"Build NFA"** 按钮：系统会使用 Thompson 算法解析你录入的正则表达式并构造出 NFA。输出框中将用文本表格形式呈现 NFA 的状态转换表。
2. 点击 **"Build DFA"** 按钮：系统将采用子集构造法（Subset Construction）将刚才的 NFA 转换为 DFA，并在右侧输出 DFA 状态表。
3. 点击 **"Min DFA"** 按钮：系统利用 Hopcroft/Moore 划分算法将 DFA 最小化，合并等价状态，并输出最终的 MinDFA 状态表。

*(注意：必须按顺序依次生成 NFA -> DFA -> Min DFA，不可跳过。)*

### 第三步：生成词法分析器源代码

1. 确保成功构建了最小化 DFA（Min DFA）后，点击 **"Gen Lexer Code"**。
2. 系统将使用“表驱动法”（对应任务书中的“转换方法一”）在项目目录下立刻生成一份 C++ 源码：`lexer.cpp`。
3. 右侧面板会完整展示出该源码的具体内容。其中包括了自动机的状态迁移静态表、终结符编码数组以及相应的关键字拦截识别逻辑。

### 第四步：系统测试与单词编码输出

为了测试生成的词法分析器（`lexer.cpp`），需要提供一段目标语言的源代码（如 `sample.tny` 或者你自行编写的 `test.tny`）。

1. 点击 **"Load Source File"** 按钮，选择你想测试的源代码文件（例如 `.tny` 或 `.c` 文件）。
2. 点击 **"Compile & Run"** 按钮。
   - 程序在后台会调用本地的编译命令：`g++ -O2 lexer.cpp -o lexer.exe`。
   - 编译成功后，将会执行 `lexer.exe "你的测试文件路径"`。
   - 分析结果（Token流序列记录，如 `103 101 "a" 204 ...`）会被保存到与源代码同名的 `.lex` 文件中（例如 `sample.lex` ）。
   - 程序随后读取生成的 `.lex` 结果，及其终端的标准输出，在右侧 Output 控制面板进行展示，用户可直接查看该单词编码。

## 4. 特殊注意事项

1. **转义字符支持**：请使用 `\` 进行转义操作（例如加号写为 `\+`，星号写为 `\*`）。转义字符在正则中被正确支持。
2. **注释的处理**：通过正则中包含 "comment" 字眼的规则会被词法分析器匹配为合法词法单元，但在输出到 `.lex` 文件之前，生成器会自动把它们过滤丢弃，从而符合任务要求中“注释不需要输出单词编码”的规定。
3. **中文路径**：如果路径含有中文字符，`g++` 可能会偶尔引发编码找不到文件的问题，建议尽量把测试文件放在英文目录下，不过本程序对常规的中文操作已经做了宽容处理。

# 编译原理课程设计 - LALR(1) 语法分析器生成及分析工具 (Task 2) 使用说明书

## 1. 软件介绍
本软件是为满足《编译原理课程设计》第二项任务要求而开发的“语法分析器自动生成及分析工具”（Parser Generator / ParserGen）。
该程序完全采用 C++ 及原生的 Windows Win32 API 编写，提供美观、直观的三栏式纯图形化窗口界面（GUI）。
本软件主要功能包括：
1. **文法输入与解析**：以文本形式或在界面上直接编辑并输入 BNF 文法规则，默认第一条非终结符为文法开始符号，并使用 `@` 表示空串（$\epsilon$）。
2. **First/Follow 集合计算**：一键求出文法中所有非终结符的 First 集合和 Follow 集合，以清晰的表项进行排版展现。
3. **自动机（DFA）生成**：支持自动构造出完整的 **LR(1) 自动机**以及进行等价合并后的 **LALR(1) 自动机**，以状态转移表格及项集表形式呈现点和边。
4. **LALR(1) 分析表计算**：根据 LALR(1) 状态机自动计算出 LALR(1) 分析表（Action & Goto 表），在分析表中会自动识别冲突（Shift/Reduce 或 Reduce/Reduce）并给出高能警告。
5. **词法输出解析与语法分析**：可导入或拷贝第一阶段（LexerGen）输出的目标语言 `.lex` 单词编码文件，支持高度可自定义的 **Token ID 映射编辑器**。
6. **一步到位语法树生成**：通过 LALR(1) 移进-归约算法，现场执行语法分析，输出极其详尽的**步进式语法分析过程日志**（包括每一步的步骤号、状态栈、符号栈、剩余输入缓存区、所做出的移进/归约动作），并最终在控制台与输出框中生成以树状缩进呈现的**完整语法树**。

---

## 2. 软件设计架构与可扩展性
本程序核心逻辑经过模块化设计，具备卓越的底层性能优势与可扩展空间：
* **Grammar（文法处理模块）**：所有文法符号在解析时均被转化为整型 ID 存储及哈希映射。这一设计大大超越了传统使用字符串直接比对的做法，大幅降低了 First/Follow 集合迭代计算时的内存开销。
* **LALRAutomata（LALR 自动机模块）**：对状态、LR(1) 项目（Item）、Lookahead 进行了紧凑的面向对象封装。如果后续需要针对极大规模的工业级文法进一步优化，其底层的 `build_lalr()` 函数与状态转移接口可无缝修改为引入著名的 *DeRemer & Pennello* 高效 Lookahead 传播算法，且完全不会破坏主界面及引擎的其他模块。
* **ParserEngine（语法分析引擎模块）**：在归约时不仅完成了状态转移，更通过多叉树结构动态构建出了内存语法树结构，可直接用于后续的代码生成或中间代码转换扩展。

---

## 3. 环境与编译要求
1. **操作系统**：Windows Vista / 7 / 8 / 10 / 11 操作系统。
2. **编译器**：本机 `PATH` 环境变量中需配有 `g++` 编译器（MinGW / MSYS2）。
3. **编译执行**：双击 `ParserGen/build.bat` 批处理文件，它将自动在本地环境中完成源码的极速优化编译，并在同级目录下输出 `ParserGenerator.exe` 可执行程序。

---

## 4. 主界面操作指南
双击运行 `ParserGenerator.exe` 后，您将看到一个美观的三栏式主控制台界面：

### 4.1 左栏：文法规则控制区 (BNF Grammar)
* **文本编辑框**：用于直接编辑或查看当前的 BNF 文法。
  - **文法书写规范**：
    - 使用 `->` 表示产生式箭头，如 `E -> E + T`。
    - 产生式的右部可以使用 `|` 符号隔开多个分支。
    - 用大写或小写字母词组代表非终结符和终结符，但必须保证词意一致。
    - `@` 特殊字符代表空串（$\epsilon$），在移进归约时它会被作为空转换处理。
    - `$` 符号代表输入结束标志。
* **"Load BNF File" 按钮**：一键打开文件管理器，支持导入外置的文法规则文本文件（如 `TINY语言的所有语法.txt` 或 `mini-c的所有语法.txt`）。
* **"First/Follow Sets" 按钮**：实时计算文法内所有非终结符的 First 集合和 Follow 集合，结果在右栏输出。
* **"View LR(1) DFA" 按钮**：构造并全面列出 LR(1) 状态机的每一个状态节点、内含的 LR(1) 核心及展望字符项目集、以及 GOTO 转换边。
* **"View LALR(1) DFA" 按钮**：通过合并核心等价的 LR(1) 状态，生成更小巧高效的 LALR(1) 自动机状态集。
* **"View LALR Table" 按钮**：生成并打印格式完美的 LALR(1) 语法分析表。如果文法中存在二义性冲突，输出区顶部会特别标示出 `!!! WARNING: Grammatical conflicts detected`，并自动按“移进优先于归约”的标准规避规则完成表格构建。

### 4.2 中栏：映射自定义与词法输入 (Token Map & .lex)
由于语法分析器的工作核心是处理单词的“单词编码”（Token ID），但 BNF 文法规则中书写的是如 `number`、`identifier`、`if` 等直观的终结符名。为了将两阶段串联起来，ParserGen 提供了业内领先的 **Token ID 映射编辑器**：
* **"Token ID Map (Customize)" 编辑框**：
  - 每行填写一个映射规则，格式为：`[TokenID] [GrammarTerminalSymbolName]`。
  - **默认 TINY 语言映射** 已在启动时预填。例如：
    - `100 number` （将词法分析器输出的100号 Token 识别为语法文法中的 `number`）
    - `101 identifier` （将101号 Token 识别为 `identifier`）
    - `103 if` （将103号 Token 识别为 `if` 关键字）
    - `201 +` （将201号 Token 识别为加号操作符）
  - 用户可随时在此编辑框内增删、自定义改动，以支持 Mini-C 等不同语言词法器的适配。
* **"Reset Map to TINY" 按钮**：快速一键恢复到默认的 TINY 单词编码集。
* **"Source Token Stream (.lex)" 编辑框**：
  - 复制、粘贴或在此框内手工录入需要分析的 `.lex` 文件内容。
  - **数据格式**：支持形如 `103  101 "x"  206  100 0 ...` 的单词和属性流（与第一阶段 LexerGen 输出的 sample.lex 结构无缝兼容）。
* **"Load .lex File" 按钮**：用于快速从磁盘中导入生成的 `.lex` 文件。
* **"Run LALR Parser" 按钮**：最核心的按钮！一键根据当前的 BNF 规则、LALR 分析表与映射，解析当前的 Token 流，执行彻底的移进-归约算法，在右栏大控制台中画出完整的语法树，并追溯全套的分析流日志。

### 4.3 右栏：控制台/结果展现面板 (Console Panel)
* **大输出显示框**：以超宽视野的 Consolas 字体清晰展示每一次动作的所有计算细节。分析过程日志（Log）包含：
  - **Step**：动作步骤序号。
  - **State Stack**：移进-归约的状态栈变动。
  - **Symbol Stack**：符号栈内归约过程所涵盖的全部文法项。
  - **Input Buffer**：剩余输入单词缓冲区的当前切片。
  - **Action**：所做出的判断，如：`Shift to state 5` 或 `Reduce by: statement -> read identifier`。
  - **树状缩进语法树**：利用 ASCII 字符渲染的语法树层次分明地展现出了词法到高层文法的完整派生路径。

## 5. 快速测试流程 (针对 TINY 语言)
1. **加载语法**：点击左下角 **"Load BNF File"**，选择 `TINY语言的所有语法.txt`（程序在启动时已经预先载入了标准 TINY 语法，您可以直接使用它进行测试）。
2. **分析测试**：点击左下角 **"First/Follow Sets"** 以校验 First/Follow 集构建情况。也可以点击 **"View LALR Table"** 观摩状态表转移结构。
3. **加载 Token 文件**：点击中下角 **"Load .lex File"**，选择由 Task 1 的词法分析器生成的 `sample.lex`（或者把该文件内容直接粘贴到“Source Token Stream”编辑框中）。
4. **一键生成语法树**：点击中下角右侧的 **"Run LALR Parser"** 按钮。
5. **查看结果**：在右侧控制台中下拉即可饱览精妙绘制的缩进语法树和完美步进式状态转换表格！

---

## 6. 特殊规则与说明
1. **空产生式处理**：如果您的文法中有 `A -> @`（如 TINY 或 Mini-C 的空语句或空参数），系统会自动识别并在归约 `A -> @` 时弹起状态栈中的 `0` 个状态，同时自动向语法树中补充一个代表 `@` (空值叶子) 的展示分支，确保语法树完整美观。
2. **展望符与结束标记**：整个系统严格按照标准的 LALR(1) 理论在状态项目右侧使用 `, [lookahead]` 控制展望行为，且在 Token 链尾部默认自动追加 `$ (END_MARKER)` 作为全局接受判定，具有极强的鲁棒性。
